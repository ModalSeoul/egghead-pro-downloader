            <p>Hello and welcome to our lesson on JSON Web Tokens also known as JWTs. What we&#39;re going to do is create a JWT using the node command line. The way I&#39;m going to do this is by creating each of the three pieces of a JWT -- the header, the payload, and the key -- sign the key, and then show you all how to debug a JWT.</p>

<p>First, let&#39;s start with our header. Our header is a JSON object with two claims. Claims are a fancy word for JSON key value pairs, and the two we&#39;re putting here are typ which stands for type, and alg which stands for algorithm which is what we&#39;ll use to sign the key.</p>

<p>Now that we have our header object, we need to turn it into a base64 encoded string. We&#39;re going to do that by placing it in a buffer with JSON.stringify, our header, and then to string that bugger with base64.</p>

<p>We have our header. What we&#39;re going to do is create our payload. When I talk about claims, I mentioned that there are fancy words for JSON key pairs, but there are several types of claims.</p>

<p>For instance, there are public claims which are registered in the specification in under a registry, there are reserved claims like iat which stands for issued at which are part of the specification, or issuer also part of the specification, and then there&#39;s private claims which are claims you can write yourself. Things like user name.</p>

<p>These can have more than three characters whereas public and reserve tend to have three characters. Now that we have our payload, we&#39;re going to turn it into a base64 encoded string just like we did with the previous one.</p>

<p>Now that we have our header, and our payload, let&#39;s talk about creating the key. There&#39;s a little bit more to creating the key than the other two items. What we need to do is first create our base key which is our header. Can catenate it with a period, can catenate it with our payload.</p>

<p>We need to sign the key. To do that, we&#39;re going to use the crypto library for node. We&#39;re going to create a signature, and it&#39;ll be a crypto.createHmac. The first thing we&#39;re going to pass it is sha256 which is the algorithm we mentioned in the header. The second thing we&#39;re going to pass it is our secret.</p>

<p>This secret should be shared between the server, and whichever parties you want to authenticate the token with because that&#39;s how they&#39;ll know that the identity is verified. Now that we have our signature Hmac, we&#39;re going to update it with our key.</p>

<p>Finally, to create the finalized version of our key, we&#39;re going to digest this signature with the format of base64. We have all of the pieces of our JSON Web token, so let&#39;s assemble it. All the JSON token is, is our header, plus the period, plus the payload, plus another period, plus the signed digested key.</p>

<p>Let&#39;s take a look, and now we&#39;ve got our JSON token. I&#39;m going to copy that, and I&#39;m going to next show you how to debug this JSON Web token using the jwt.io website. This is the jwt.io website. If you scroll down just a little bit, you&#39;ll find the debugger. What we&#39;re going to do is we&#39;re going to paste our newly made JWT into this debugger.</p>

<p>As you can see on the right, it decodes all the base64 encoded information, and shows it here in the header, and payload. It talks a little bit about our signature. <p>I&#39;m going to change it, and as you can see, it now says signature verified. That&#39;s how you make sure that the JWTs that are sent to your server are from who they say they are, is by verifying that the key was signed in the correct way.</p>

<p>You can also sign keys asymmetrically with RSA 56 public private key pairs. To learn even more about JWTs, I highly recommend the jwt.io website.           <div class="well padded white-background">
            <p>Hello and welcome to our lesson on JSON Web Tokens also known as JWTs. What we&#39;re going to do is create a JWT using the node command line. The way I&#39;m going to do this is by creating each of the three pieces of a JWT -- the header, the payload, and the key -- sign the key, and then show you all how to debug a JWT.</p>

<p>First, let&#39;s start with our header. Our header is a JSON object with two claims. Claims are a fancy word for JSON key value pairs, and the two we&#39;re putting here are typ which stands for type, and alg which stands for algorithm which is what we&#39;ll use to sign the key.</p>

<p>Now that we have our header object, we need to turn it into a base64 encoded string. We&#39;re going to do that by placing it in a buffer with JSON.stringify, our header, and then to string that bugger with base64.</p>

<p>We have our header. What we&#39;re going to do is create our payload. When I talk about claims, I mentioned that there are fancy words for JSON key pairs, but there are several types of claims.</p>

<p>For instance, there are public claims which are registered in the specification in under a registry, there are reserved claims like iat which stands for issued at which are part of the specification, or issuer also part of the specification, and then there&#39;s private claims which are claims you can write yourself. Things like user name.</p>

<p>These can have more than three characters whereas public and reserve tend to have three characters. Now that we have our payload, we&#39;re going to turn it into a base64 encoded string just like we did with the previous one.</p>

<p>Now that we have our header, and our payload, let&#39;s talk about creating the key. There&#39;s a little bit more to creating the key than the other two items. What we need to do is first create our base key which is our header. Can catenate it with a period, can catenate it with our payload.</p>

<p>We need to sign the key. To do that, we&#39;re going to use the crypto library for node. We&#39;re going to create a signature, and it&#39;ll be a crypto.createHmac. The first thing we&#39;re going to pass it is sha256 which is the algorithm we mentioned in the header. The second thing we&#39;re going to pass it is our secret.</p>

<p>This secret should be shared between the server, and whichever parties you want to authenticate the token with because that&#39;s how they&#39;ll know that the identity is verified. Now that we have our signature Hmac, we&#39;re going to update it with our key.</p>

<p>Finally, to create the finalized version of our key, we&#39;re going to digest this signature with the format of base64. We have all of the pieces of our JSON Web token, so let&#39;s assemble it. All the JSON token is, is our header, plus the period, plus the payload, plus another period, plus the signed digested key.</p>

<p>Let&#39;s take a look, and now we&#39;ve got our JSON token. I&#39;m going to copy that, and I&#39;m going to next show you how to debug this JSON Web token using the jwt.io website. This is the jwt.io website. If you scroll down just a little bit, you&#39;ll find the debugger. What we&#39;re going to do is we&#39;re going to paste our newly made JWT into this debugger.</p>

<p>As you can see on the right, it decodes all the base64 encoded information, and shows it here in the header, and payload. It talks a little bit about our signature. However, as you can see, it says invalid signature. That&#39;s because the secret here is not the secret we used to sign our key.</p>

<p>I&#39;m going to change it, and as you can see, it now says signature verified. That&#39;s how you make sure that the JWTs that are sent to your server are from who they say they are, is by verifying that the key was signed in the correct way.</p>

<p>You can also sign keys asymmetrically with RSA 56 public private key pairs. To learn even more about JWTs, I highly recommend the jwt.io website. We have a lot of information there, and you can definitely read it a little more easily than the specification. Though I would definitely recommend reading RFC 7519 to learn more.</p>

<p>Thanks for listening.</p>

            <p>Hello and welcome to our lesson on JSON Web Tokens also known as JWTs. What we&#39;re going to do is create a JWT using the node command line. The way I&#39;m going to do this is by creating each of the three pieces of a JWT -- the header, the payload, and the key -- sign the key, and then show you all how to debug a JWT.</p>

<p>First, let&#39;s start with our header. Our header is a JSON object with two claims. Claims are a fancy word for JSON key value pairs, and the two we&#39;re putting here are typ which stands for type, and alg which stands for algorithm which is what we&#39;ll use to sign the key.</p>

<p>Now that we have our header object, we need to turn it into a base64 encoded string. We&#39;re going to do that by placing it in a buffer with JSON.stringify, our header, and then to string that bugger with base64.</p>

<p>We have our header. What we&#39;re going to do is create our payload. When I talk about claims, I mentioned that there are fancy words for JSON key pairs, but there are several types of claims.</p>

<p>For instance, there are public claims which are registered in the specification in under a registry, there are reserved claims like iat which stands for issued at which are part of the specification, or issuer also part of the specification, and then there&#39;s private claims which are claims you can write yourself. Things like user name.</p>

<p>These can have more than three characters whereas public and reserve tend to have three characters. Now that we have our payload, we&#39;re going to turn it into a base64 encoded string just like we did with the previous one.</p>

<p>Now that we have our header, and our payload, let&#39;s talk about creating the key. There&#39;s a little bit more to creating the key than the other two items. What we need to do is first create our base key which is our header. Can catenate it with a period, can catenate it with our payload.</p>

<p>We need to sign the key. To do that, we&#39;re going to use the crypto library for node. We&#39;re going to create a signature, and it&#39;ll be a crypto.createHmac. The first thing we&#39;re going to pass it is sha256 which is the algorithm we mentioned in the header. The second thing we&#39;re going to pass it is our secret.</p>

<p>This secret should be shared between the server, and whichever parties you want to authenticate the token with because that&#39;s how they&#39;ll know that the identity is verified. Now that we have our signature Hmac, we&#39;re going to update it with our key.</p>

<p>Finally, to create the finalized version of our key, we&#39;re going to digest this signature with the format of base64. We have all of the pieces of our JSON Web token, so let&#39;s assemble it. All the JSON token is, is our header, plus the period, plus the payload, plus another period, plus the signed digested key.</p>

<p>Let&#39;s take a look, and now we&#39;ve got our JSON token. I&#39;m going to copy that, and I&#39;m going to next show you how to debug this JSON Web token using the jwt.io website. This is the jwt.io website. If you scroll down just a little bit, you&#39;ll find the debugger. What we&#39;re going to do is we&#39;re going to paste our newly made JWT into this debugger.</p>

<p>As you can see on the right, it decodes all the base64 encoded information, and shows it here in the header, and payload. It talks a little bit about our signature. However, as you can see, it says invalid signature. That&#39;s because the secret here is not the secret we used to sign our key.</p>

<p>I&#39;m going to change it, and as you can see, it now says signature verified. That&#39;s how you make sure that the JWTs that are sent to your server are from who they say they are, is by verifying that the key was signed in the correct way.</p>

<p>You can also sign keys asymmetrically with RSA 56 public private key pairs. To learn even more about JWTs, I highly recommend the jwt.io website. We have a lot of information there, and you can definitely read it a little more easily than the specification. Though I would definitely recommend reading RFC 7519 to learn more.</p>

<p>Thanks for listening.</p>
